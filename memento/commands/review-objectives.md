# 프로젝트 체크포인트 (project-checkpoint)

KR1 프로세스 준수율 트래킹 파일을 업데이트한다. Jira 에픽 상태를 동기화하고, 에픽별 체크포인트 충족 여부를 인터뷰한 뒤, 메트릭을 집계하여 트래킹 파일을 갱신한다.

## 상수

| 이름 | 값 |
|------|-----|
| 트래킹 파일 상대경로 | `10 Projects/2026 Imagoworks/26 OKR/kr1-tracking.md` |
| Jira 프로젝트 | CND |
| Phase 에픽 | CND-1121, CND-1122, CND-1123, CND-1125 |
| Feature 에픽 | CND-1126, CND-1127, CND-1128, CND-1129 |
| 체크포인트 종류 | 4개: 요구사항확정+유저스토리, 일정산정+설계리뷰, 구현완료, 검증완료(SDVR Pass) |
| 업데이트 주기 | 3일 |

## 워크플로우

### Step 1: 설정 로드 + 트래킹 파일 읽기

`~/.claude/plugins/data/memento-cc-plugins/config.md` 파일을 읽는다.

| 케이스 | 처리 |
|--------|------|
| 파일 존재 | `vault`, `display_name_ko`, `display_name_en`, `initials`, `user_id`, `nickname`, `email`, `aliases`, `atlassian_account_id` 값을 로드 |
| 파일 없음 | "설정이 없습니다. `/memento:setup`을 먼저 실행해주세요." 안내 후 중단 |

**사용자 식별 컨텍스트 주입**:

식별 필드 중 비어있지 않은 값이 있으면 내부 컨텍스트에 2-3줄 블록으로 고정:

```
사용자 식별:
- 표시 이름: "<DISPLAY_KO>" / "<DISPLAY_EN>" (이니셜: <INITIALS>, 닉네임: <NICKNAME>)
- 아이디: <USER_ID> (aliases: <ALIASES>)
- 이메일: <EMAIL>
- Jira accountId: <ATLASSIAN_AID>

Feature 에픽의 assignee가 이 사용자와 일치하면 "본인 담당" 으로 분류한다.
```

빈 값 필드는 괄호/줄 생략. 모든 식별 필드가 빈 값이면 블록 자체 생략.

1. Obsidian vault 경로를 파악한다:
   ```bash
   obsidian vaults verbose
   ```
   vault 이름에 해당하는 경로를 추출.

2. 트래킹 파일을 읽는다 (Read 도구 사용):
   - 경로: `{vault_path}/10 Projects/2026 Imagoworks/26 OKR/kr1-tracking.md`

| 케이스 | 처리 |
|--------|------|
| 파일 없음 | "KR1 트래킹 파일이 없습니다. 경로: `{expected_path}`" 안내 후 중단 |
| 파일 있음 | 아래 정보를 파싱 |

3. 파싱 대상:
   - 블록인용 메타데이터: `> 다음 업데이트:`, `> 마지막 업데이트:`, `> 상태:`
   - `## Dashboard` 섹션: 전체 준수율, 평균 일정 오차, 체크포인트별 달성률 테이블, 활성 리스크
   - `## Phase 에픽 현황` 섹션: Phase 에픽 테이블
   - `## Feature 에픽별 프로젝트 체크포인트` 섹션: 에픽별 Jira 상태, 체크포인트 테이블 (충족/날짜/근거/비고), 일정 오차
   - `## Change Log` 섹션: 기존 로그 (추가용)

### Step 2: Jira 동기화

1. 전체 에픽 상태를 조회한다:

   JQL: `project = CND AND key in (CND-1121, CND-1122, CND-1123, CND-1125, CND-1126, CND-1127, CND-1128, CND-1129)`

   `mcp__plugin_atlassian_atlassian__searchJiraIssuesUsingJql` 도구를 사용한다.
   각 이슈에서 추출: `key`, `fields.status.name`, `fields.summary`.

2. Feature 에픽(CND-1126~1129)의 상세 정보를 조회한다:

   `mcp__plugin_atlassian_atlassian__getJiraIssue` 도구로 각 에픽을 조회한다.
   추가 추출: `fields.assignee.accountId`, `fields.assignee.displayName`, `fields.duedate`.

3. 각 Feature 에픽에 **`is_self` 불린 필드**를 계산한다 (본인 담당 여부):

   - **1순위**: `atlassian_account_id`가 비어있지 않고 `assignee.accountId`와 정확 일치 → `true`
   - **2순위**: `display_name_ko`가 비어있지 않고 `assignee.displayName`과 정확 일치 → `true`
   - **3순위**: `display_name_en`가 비어있지 않고 `assignee.displayName`과 대소문자 무시 + 공백 제거 비교 일치 → `true`
   - **4순위**: `nickname`이 비어있지 않고 `assignee.displayName`과 포함 관계 → `true`
   - 모두 실패 → `false`

   accountId 우선 매칭으로 동명이인/한영 혼용 에지 케이스를 해소한다.

4. 조회 결과를 정리한다:
   - Phase 에픽: key → Jira 상태 매핑
   - Feature 에픽: key → Jira 상태, 담당자, accountId, `is_self`, 마감일 매핑

| 케이스 | 처리 |
|--------|------|
| 동기화 성공 | Step 3에서 최신 Jira 데이터 사용 |
| 동기화 실패 (타임아웃, 인증 오류 등) | "⚠ Jira 동기화 실패: {error}. 기존 데이터로 진행합니다." 경고 출력 후 트래킹 파일의 기존 Jira 상태를 그대로 사용 |

### Step 3: 체크포인트 리뷰 인터뷰

에픽 단위로 배치 질문한다 (AskUserQuestion × Feature 에픽 수).

**순회 순서**: `is_self == true` 에픽을 **먼저** 처리한 뒤, 나머지 에픽을 원래 순서대로 처리한다. 본인 담당 에픽은 약속 문제 리스크(리더십 #5/#6/#7)와 직결되므로 우선 리뷰한다.

**각 Feature 에픽에 대해**:

1. 현재 상태를 제시한다 (`is_self == true`면 헤더 끝에 ` — 🔸본인 담당` 접미사 추가):

```
### {epic_key} {epic_title} (@{assignee}) — Jira: {jira_status}{본인_담당_접미사}

현재 체크포인트:
| # | 체크포인트 | 충족 | 날짜 | 근거 |
|---|----------|:----:|------|------|
| 1 | 요구사항 확정 + 유저스토리 변환 | {O/-} | {date} | {evidence} |
| 2 | 일정 산정 + 설계 리뷰 완료 | {O/-} | {date} | {evidence} |
| 3 | 구현 완료 | {O/-} | {date} | {evidence} |
| 4 | 검증 완료 (SDVR Pass) | {O/-} | {date} | {evidence} |

변경사항이 있나요?
- 예시: "1 충족, 4/7, Jira 유저스토리 완료"
- 복수: "1 충족, 4/7, 유저스토리 완료 / 2 충족, 4/9, 설계 리뷰 통과"
- 변경 없음: "없음"
```

2. 사용자 응답을 파싱한다:
   - `"없음"` → 해당 에픽 변경 없음, 다음 에픽으로
   - `"{번호} 충족, {날짜}, {근거}"` → 해당 체크포인트를 충족(`O`)으로 변경, 날짜와 근거 기록
   - 복수 항목은 `/` 또는 줄바꿈으로 구분

| 케이스 | 처리 |
|--------|------|
| Jira 상태가 "백로그" + 모든 체크포인트 미충족 | "백로그 상태입니다. 건너뛸까요?" 제안. 사용자가 "예" → 스킵 |
| 이미 모든 체크포인트 충족(`O`) | "모든 체크포인트가 이미 충족입니다." 안내 후 스킵 |

### Step 4: 일정 오차 수집

Step 3에서 CP3(구현 완료) 또는 CP4(검증 완료)가 **새로 충족**된 에픽만 대상으로 한다.

| 케이스 | 처리 |
|--------|------|
| 해당 에픽 없음 | 이 Step 전체 스킵 |
| 해당 에픽 있음 | 에픽별로 아래 질문 |

AskUserQuestion:

```
{epic_key}의 구현→검증 일정 정보를 업데이트해주세요:
- 구현→검증 계획일: (예: 2026-05-15)
- 구현→검증 실제일: (예: 2026-05-17, 또는 "미정")
```

- "미정" → 실제일 비워둠
- 양쪽 날짜가 있으면 오차 = `abs(실제일 - 계획일)` (일 단위)

### Step 5: 메트릭 집계 + 리스크 감지

#### 메트릭 계산

1. **전체 준수율**:
   - 모든 Feature 에픽의 충족(`O`) 체크포인트 수를 합산
   - 전체 = Feature 에픽 수 × 4 (현재 16)
   - 준수율 = `(충족 수 / 전체) × 100` → 소수점 첫째 자리 반올림
   - 표시: `N% (X/16)` — 목표: 80%

2. **평균 일정 오차**:
   - 계획일/실제일 모두 있는 에픽만 대상
   - 각 에픽 오차(절대값)의 평균
   - 해당 에픽 없으면 `-`
   - 표시: `N일` — 목표: 2일 이내

3. **체크포인트별 달성률** (Dashboard 테이블용):
   - 각 체크포인트 종류별로, 4개 Feature 에픽 중 충족 수 카운트
   - 표시: `X/4`

#### 리스크 감지 규칙

| 조건 | 리스크 메시지 |
|------|-------------|
| CP1 미충족 + Jira 상태가 "진행 중" 계열 | "{epic_key}: 요구사항 미확정 상태에서 개발 진행 중" |
| `is_self == true` + CP1/CP2 미충족 + Jira 상태 "진행 중" 계열 | "{epic_key}: 본인 담당 에픽의 기획·설계 체크포인트 미충족 상태에서 진행 — 약속 리스크" |
| 일정 오차 > 2일 | "{epic_key}: 일정 오차 {N}일 (목표 2일 이내 초과)" |
| CP 미충족인데 다음 단계 Jira 상태 | "{epic_key}: 체크포인트 미충족 상태에서 다음 단계 진행" |
| 전체 준수율 < 80% + 현재 6월 이후 | "전체 준수율 {N}% — 목표(80%) 달성 위험" |
| 해당 없음 | "(없음)" |

### Step 6: 트래킹 파일 갱신

Edit 도구로 섹션별 정밀 수정한다. **Write로 전체 파일을 덮어쓰지 않는다.**

#### 6-1. 블록인용 메타데이터

`다음 업데이트`일 계산:
- 오늘 + 3일
- 토요일이면 → 월요일 (+2일)
- 일요일이면 → 월요일 (+1일)

```
> 다음 업데이트: {YYYY-MM-DD} ({요일})
> 마지막 업데이트: {오늘 YYYY-MM-DD}
> 상태: {새 상태 — 예: "진행 중 — 준수율 25%"}
```

#### 6-2. Dashboard 섹션

- 전체 준수율 행 갱신
- 평균 일정 오차 행 갱신
- 체크포인트별 달성률 테이블 갱신
- 활성 리스크 목록 갱신 (Step 5 결과)

#### 6-3. Phase 에픽 현황

- Phase 에픽 테이블의 Jira 상태 열 갱신 (Step 2 결과)
- `마지막 Jira 동기화:` 날짜 갱신

#### 6-4. Feature 에픽별 프로젝트 체크포인트

각 Feature 에픽 섹션에서:
- `Jira 상태:` 행 갱신
- 체크포인트 테이블 행 갱신 (충족, 날짜, 근거, 비고)
- 일정 오차 섹션 갱신 (계획일, 실제일, 오차)

#### 6-5. Change Log

기존 Change Log의 **맨 위**에 오늘 항목을 추가한다:

```markdown
### {YYYY-MM-DD}
- Jira 동기화 완료
- {변경된 체크포인트 요약, 예: "CND-1126 CP1 충족 처리 (4/7)"}
- {리스크 변경 요약, 예: "리스크 추가: CND-1127 요구사항 미확정"}
- 준수율: {N}% ({X}/{total})
```

변경사항이 없으면:
```markdown
### {YYYY-MM-DD}
- Jira 동기화 완료
- 체크포인트 변경 없음
- 준수율: {N}% ({X}/{total})
```

### Step 7: 요약 리포트

```
---
## KR1 체크포인트 업데이트 완료

**준수율**: {N}% ({X}/16) — 목표 80%
**평균 일정 오차**: {N일 또는 "-"} — 목표 2일 이내
**다음 업데이트**: {YYYY-MM-DD} ({요일})

### 변경사항
- {변경 목록, 없으면 "변경 없음"}

### 활성 리스크
- {리스크 목록, 없으면 "(없음)"}

파일: {트래킹 파일 절대 경로}
---
```

## Do / Don't

| Do | Don't |
|----|-------|
| 에픽 단위 배치 질문으로 효율적 인터뷰 | 체크포인트 16개를 하나씩 질문 |
| Jira 실패 시 기존 데이터로 계속 진행 | Jira 오류로 전체 중단 |
| Edit 도구로 섹션별 정밀 수정 | Write로 전체 파일 덮어쓰기 |
| 다음 업데이트일 주말 회피 (→ 월요일) | 토/일에 리마인드 |
| 변경 없는 에픽은 빠르게 스킵 | 변경 없어도 전체 리뷰 강요 |
| Change Log에 변경사항 + 준수율 기록 | 변경 이력 없이 숫자만 갱신 |
| 리스크 규칙 기반 자동 감지 | 리스크를 임의로 판단 |
| 일정 오차는 CP3/CP4 신규 충족 시에만 질문 | 매번 모든 에픽의 일정 오차를 질문 |
