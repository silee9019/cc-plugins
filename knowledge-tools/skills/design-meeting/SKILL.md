# Design Meeting

정의가 명확하면 이름과 결정은 자연스럽게 따라온다. 정의가 모호하면 어떤 투표도 수렴하지 않는다.

단일 분석자의 병렬 자문 수집이 아니라, **페르소나 에이전트들의 상호작용 라운드 + facilitator 주재**로 정의를 쌓아 올린다.

## 핵심 원칙

- **Priority-0 의도 정합성 우선** — 기술적 분석 전에 원안(사용자 요청 + 기존 문서)과 진행 방향이 정합한지 먼저 검증한다. 이탈 시 즉시 복구 후 진행.
- **상호작용 > 병렬 설문** — Step 2, Step 4는 라운드 기반 다자 토론. 각 페르소나는 이전 발언 전체를 읽고 반박·확장·수렴한다.
- **리서치/검증은 독립 병렬 유지** — hallucination 방어 불변식. Step 3a 리서치와 3b 검증은 반드시 다른 에이전트로 분리.
- **정의가 척추** — 모든 단계가 "이것은 무엇인가?"를 중심으로 전개. 이름/결정은 정의에서 도출.
- **facilitator 책임** — 메인 세션이 사회자 역할을 겸한다. 발언권, 라운드 종료, 교착 해소를 주재.
- **내부 Task ID 축약 단독 사용 금지** — 산출물(결정 기록, 대화)에서 `T1`, `CP1`, `KR1`, 에픽 내부 순번 같은 축약을 한 문서 내 첫 출현 시 단독 사용하지 않는다. 풀어쓰거나 괄호 병기. 이후 반복은 단독 허용. 상세: 저장소 CLAUDE.md "사용자 대면 출력 규칙".

## 페르소나 시스템

### 코어 멤버 (항상 참여)

| ID | 역할 | 분석 축 | 프롬프트 파일 |
|----|------|---------|-------------|
| `ontologist` | 온톨로지스트 | 본질 (essence) | `personas/core/ontologist.md` |
| `sw-expert` | SW공학 전문가 | 체계 (taxonomy) | `personas/core/sw-expert.md` |
| `user-proxy` | 사용자 대리인 | 경험 (experience) | `personas/core/user-proxy.md` |

### 벤치 멤버 (주제 유형별 자동 선택)

자동 선택 가이드: `reference/persona-selection.md` 참조. `bench_policy` 플래그로 강도 조절.

| 분석 유형 | 기본 벤치 셋 |
|----------|-------------|
| 용어/네이밍 | linguist, backend-dev, frontend-dev, designer |
| 개념 정의 | linguist, tech-writer, architect |
| 기술 선택 | architect, backend-dev, devops |
| 아키텍처 | architect, backend-dev, devops, security |
| API 설계 | architect, backend-dev, frontend-dev, tech-writer |
| 데이터 모델 | architect, data-engineer, backend-dev |
| UI/UX 결정 | designer, frontend-dev, pm |
| 메타-설계 (스킬/프로세스/프로토콜) | architect, tech-writer, pm, qa-engineer |

AskUserQuestion으로 벤치 멤버 추가/제거 확인. 주제가 복수 유형에 걸치면 유형 태그를 set으로 수집하고 벤치도 합집합으로 구성한다.

### `bench_policy` 플래그

| 값 | 동작 |
|----|------|
| `strict` | 벤치 전원 모든 라운드 참여 강제. 최대 커버리지. |
| `advisory`(기본) | 벤치는 Step 2·4에 참여. facilitator가 발언권 배분 조절 가능. |
| `off` | 코어 멤버만. 경량 결정/자기참조 주제/빠른 회전 시. |

## 실행 모드 (규모 적응)

| 모드 | 조건 | 생략 가능 |
|------|------|----------|
| `micro` | 명확한 소규모 결정, 문서화만 필요 | Step 3b 검증 축소, Step 4 단일 라운드 |
| `light`(기본) | 일반 결정, 벤치 3-4명 | Step 3a 리서치 1-2 에이전트 |
| `full` | 중대한 설계 결정, 다면 검증 필수 | 전 단계 엄격 실행 |

facilitator(메인)가 Step 0에서 판단. 필요 시 사용자에게 AskUserQuestion으로 확인.

## 워크플로우

```
[주제] → 0.파악+정합성 → 1.현상학 → 2.온톨로지(상호작용) → 3.분류학+검증(독립) → 4.다면검증(상호작용) → 5.도출+기록
                         (경험)       (본질·토론)            (위치·hallucination 방어)  (수렴·토론)              (결정)
```

### Step 0: 주제 파악 + 의도 정합성 검증

**Priority-0 원칙이 내재화된 단계.** 이 단계를 통과하지 않으면 다음 단계로 진행하지 않는다.

#### 0a. 원안 고정값 수집 (필수)

다음 출처를 모두 확인하고 "원안(intent source)"으로 고정한다:

1. **현재 대화 컨텍스트** — 사용자가 이번 세션에서 직접 표현한 요구사항·제약·선호
2. **문서화된 원안** — inbox 이슈 파일, CLAUDE.md, 관련 decision record, 이전 대화 링크
3. **프로젝트 규칙** — 레포 CLAUDE.md, 관련 컨벤션

> ⚠️ 원안은 **인용 가능한 텍스트**로 기록. "사용자가 이렇게 말했다고 추정" 금지. 출처가 없으면 "원안 없음(가정)"으로 명시.

#### 0b. 주제 유형 분류

AskUserQuestion 또는 원안 추출로 주제 유형을 확정(복수 허용):

| 유형 | 예시 | 핵심 질문 |
|------|------|----------|
| 용어/네이밍 | "이걸 영어로 뭐라 부를까?" | 이것은 무엇인가? |
| 개념 정의 | "requirement와 specification 차이가 뭐야?" | 이 둘은 어떻게 다른가? |
| 기술 선택 | "상태 관리 뭘 쓸까?" | 우리 문제의 본질은 무엇인가? |
| 아키텍처 | "이 구조가 맞나?" | 이 시스템의 본질적 제약은 무엇인가? |
| API 설계 | "이 엔드포인트 시그니처 맞아?" | 책임과 경계는 무엇인가? |
| 데이터 모델 | "이 스키마 쪼개야 하나?" | 불변식과 변화 단위는 무엇인가? |
| UI/UX 결정 | "이 플로우 맞아?" | 사용자 멘탈 모델과 정합한가? |
| 메타-설계 | "이 스킬/프로세스 괜찮아?" | 누가 언제 어떤 맥락에서 쓰는가? |

#### 0c. 재귀 주제 감지

주제가 **이 스킬 자신** 또는 **현재 세션 주도자의 직접 산출물**이면 "재귀 주제"로 태그:

- user-proxy 단독 "가정" 기반 경험 기술 **금지** — 반드시 원안 문서를 우선 근거로 사용
- Step 0d 편차 체크를 매 단계 반복

#### 0d. 의도-방향 매핑

현재 진행 방향(가설, 주제 정의)이 원안과 정합한지 체크:

| 판정 | 조건 | 조치 |
|------|------|------|
| **Clear & Aligned** | 원안 명확, 진행 방향 정합 | Step 0e로 진행 |
| **Clear but Misaligned** | 원안 명확, 진행 방향 이탈 | BLOCK. 사용자에게 이탈 지점 제시 후 재설정 |
| **Unclear** | 원안 자체가 불분명 | 회의 주최 프로토콜 (AskUserQuestion 또는 회의 지속) |

> 이 판정은 Step 1 시작 시, Step 2 종료 시, Step 4 종료 시 **최소 3회 재확인**.

#### 0e. 실행 모드 + 페르소나 확정

- 실행 모드(`micro`/`light`/`full`) 결정
- 주제 유형별 기본 벤치 셋 + `bench_policy` 확정
- facilitator가 전체 로드맵 간단 선언

**Output**: 원안(인용문), 주제+유형 set, 재귀 태그, 정합성 판정, 실행 모드, 참여 페르소나, bench_policy

### Step 1: 현상학 (Phenomenology) — "경험으로서의 대상"

**목적**: 이론적 분석 전에, 대상이 실제로 어떻게 경험되는지 수집.

**실행**: `user-proxy` 페르소나 프롬프트(`personas/core/user-proxy.md`)를 읽고 Agent로 디스패치.

**재귀 주제 가드**:
- 재귀 태그가 붙은 경우 user-proxy는 "가정"을 써내지 않는다. 원안 문서에서 인용 + 빈 정보는 "정보 부족(AskUserQuestion 필요)"으로 명시.
- 주도자(사용자) = 최종 소비자일 경우 순환 논증 방지를 위해 별도 경험 출처(inbox 이슈, 과거 대화)를 우선.

**질문 축**:
- "이걸 언제, 어떻게, 왜 쓰는가?"
- "동료에게 뭐라고 말하는가?"
- "화면에서 이걸 보면 뭐라고 부르는가?"
- "이것 없이 일하면 뭐가 불편한가?"

facilitator는 user-proxy 결과에 대해 1회 Q&A 라운드를 허용한다(불명확 시 추가 질의).

**Step 0d 재확인**: 경험 기술이 원안과 반대 방향이면 BLOCK → Step 0로 회귀.

**Output**: 사용자 경험 기술서 — 실제 사용 맥락, 자연어 표현, 일상적 호칭. 가정 항목은 모두 "가정" 태그 명시.

### Step 2: 온톨로지 분석 (Ontological Analysis) — "본질" · **라운드 기반 상호작용**

**목적**: 대상이 무엇이고 무엇이 아닌지 엄밀히 정의.

**실행 모델 변경 (design-meeting 핵심)**: 병렬 디스패치가 아닌 **라운드 기반 순차 상호작용**.

#### 라운드 프로토콜

각 라운드마다 facilitator(메인)가 다음을 수행:

1. **발언자 지정**: 다음 발언자 1명 지정 (벤치 중 라운드 주제에 가장 정합한 역할)
2. **컨텍스트 번들**: 이전 발언 전체 + 원안 요약 + Step 1 경험 기술 + 라운드 주제를 프롬프트로 구성
3. **지시**: "이전 발언을 읽고 동의/반박/확장하라. 근거를 명시하라. 새 주장은 검증 가능한 형태로 남겨라."
4. **에이전트 디스패치**: 해당 페르소나 프롬프트 Read → Agent 호출 → 결과 수집
5. **종료 판정**: 수렴(합의)/교착(반복)/신규 축 출현(계속) 중 하나로 판단

#### 라운드 구성 (기본값)

| 라운드 | 발언자 | 라운드 주제 |
|--------|--------|-----------|
| R1 | ontologist | Genus(상위 개념) + Differentia(차별점) 초안 제시 |
| R2 | sw-expert | R1에 대한 체계적 위치 매핑 (표준/학술/업계) |
| R3 | 벤치 #1 (유형별) | R1·R2 근거로 경계·관계 확장 |
| R4 | 벤치 #2 (유형별) | R1-R3에 대한 반박/보완 |
| R5 | ontologist | 합의 초안 정리 + 남은 이견 명시 |

R5에서 남은 이견이 있고 `bench_policy=strict` 또는 mode=`full`이면 R6-R7 추가 라운드.

#### 교착 해소

- 2개 라운드 연속 동일 논점 반복 → facilitator가 "이견 정리" 라운드 삽입 (사용자에게 AskUserQuestion으로 판정 위임)
- 근거 없는 주장 발견 → 해당 라운드 즉시 재실행 요청

#### 모델 배정

- `ontologist`, `sw-expert`, `architect`, `linguist`: **opus**
- 나머지 벤치, user-proxy: **sonnet**
- **haiku 사용 금지**

**Step 0d 재확인**: R5 종료 시 원안 대비 편차 체크.

**Output**:
```
정의: "A ___ is a [genus] that [differentia]"
경계: [IS / IS NOT 목록]
속성: [핵심 속성 목록]
관계: [상위/하위/인접 개념과의 관계]
라운드 기록: R1-R5 발언 요약 + 합의/이견
```

### Step 3: 분류학 + 리서치 검증 (Taxonomy & Verification) — "위치" · **독립 병렬 유지**

**목적**: Step 2의 정의를 기존 체계에 매핑하고 주장을 독립 검증.

**이 단계는 hallucination 방어 불변식** — 반드시 리서치와 검증을 **다른 에이전트**로 병렬 실행.

#### 3a. 리서치 (Agent 2-3개 병렬)

WebSearch를 사용하여:
- 표준/학계에서 이 개념에 해당하는 기존 용어 조사
- 실제 도구/업계에서 유사 개념을 어떻게 부르는지 조사
- (해당 시) DDD/도메인 언어/설계 패턴 조사

#### 3b. 교차 검증 (리서치와 반드시 다른 Agent)

리서치 결과에서 핵심 주장을 추출한 뒤:

```
[리서치 A] → 주장1, 주장2 + 출처
[리서치 B] → 주장3, 주장4 + 출처
         ↓ (핵심 주장 추출)
[검증 V1] → 주장1 독립 WebSearch 검증
[검증 V2] → 주장2 독립 WebSearch 검증  (병렬)
[검증 V3] → 주장3 독립 WebSearch 검증
[검증 V4] → 주장4 독립 WebSearch 검증
         ↓
검증 보고서: 각 주장별 ✅/⚠️/❌
```

- **주장 1개 = 검증 에이전트 1개** (병렬)
- 각 검증 에이전트는 해당 주장만 독립 WebSearch로 확인
- 출처 URL 실재 여부, 인용 정확성, 문맥 왜곡 여부 확인
- **❌ 태깅 주장은 이후 단계에서 근거로 사용 금지**

#### 검증 강도 (주제 유형별)

| 유형 | 강도 |
|------|------|
| 용어/네이밍, 개념 정의 | 표준 |
| 아키텍처, API, 데이터 모델 | 표준 + 실제 사례 1개 이상 |
| 기술 선택 | 표준 + 벤치마크/공식 문서 인용 필수 |
| 메타-설계 | 유사 도구 선례 1개 이상 |

**Output**: 검증된 분류 체계 매핑 + 기존 용어/선례 + 검증 보고서(주장별 판정)

---

### Step 4: 다면 검증 (Multi-Perspective Validation) — "수렴" · **라운드 기반 상호작용**

**목적**: 정의가 다양한 관점에서 견고한지 **상호작용 토론**으로 검증. (이름 투표가 아님)

**실행 모델**: Step 2와 동일한 라운드 기반 상호작용.

#### Step 4 strategy (주제 유형별)

Step 4의 라운드 주제를 유형에 따라 다르게 설계:

| 주제 유형 | Strategy | 라운드 주제 |
|----------|----------|-----------|
| 용어/네이밍 | name-vote | 정의 정합성 검토 → 후보 이름 도출 → 최종 투표 (1회) |
| 개념 정의 | definition-debate | 반례 수집 → 경계 재조정 → 합의 |
| 기술 선택, 아키텍처 | tradeoff-matrix | 기준 정의 → 대안별 채점 → 절충안 |
| API, 데이터 모델 | decision-debate | 대안 제시 → 제약/비용 분석 → 선택 근거 |
| 메타-설계 | risk-register | 실패 시나리오 수집 → 완화책 → 잔여 위험 |

facilitator가 주제 유형에 맞는 strategy를 선택하고 라운드 설계를 선언한다.

#### 라운드 프로토콜

Step 2와 동일. 각 라운드마다 facilitator가 발언자 지정, 컨텍스트 번들 구성, 디스패치, 종료 판정.

**검토 질문 축**:
1. "이 정의가 당신의 실무 경험/표준과 맞는가?"
2. "빠진 측면이나 반례가 있는가?"
3. "이 정의를 받아들이면, 어떤 이름/결정이 자연스러운가?"

#### 분기

- 정의에 이견 수렴 실패 → Step 2 온톨로지로 1회 회귀 (무한 루프 방지: 회귀 1회 한정)
- 정의 수렴 → 이름/결정 후보 자연 도출 → strategy별 마무리

#### Phase 분리 결정 지원

"일부는 지금 확정, 일부는 후속 재평가"가 적절한 경우 facilitator가 **Phase 1(즉시) / Phase 2(후속)** 결정 레이아웃을 선언한다.

**Step 0d 재확인**: Step 4 종료 시 원안 대비 편차 체크 (최종).

**Output**: 검증된 정의 + (필요 시) 확정된 이름/결정 + Phase 분리 결정(해당 시) + 라운드 기록

---

### Step 5: 도출 & 기록 (Derivation & Record) — "결정"

#### 5a. 최종 검증

최종 결과의 핵심 근거를 검증 에이전트로 재검증. Step 3b에서 ❌ 태깅된 근거가 재등장하지 않았는지 확인.

#### 5b. 사용자 확인

AskUserQuestion으로 최종 결과 제시:
- 확정 정의
- 확정 이름/결정 (또는 Phase 분리 결정)
- 근거 요약 (✅ 검증된 것만)
- (해당 시) 평가 결과 / trade-off / 위험
- **원안 대비 편차 요약** (Step 0d 3회 체크 결과)

#### 5c. 기록

**Obsidian vault 저장**:
- memento config (`~/.claude/plugins/data/memento-cc-plugins/config.md`)에서 vault 경로 읽기
- `templates/decision-record.md` 템플릿 사용
- 저장 위치: `{vault}/12 Records/decisions/YYYY-MM-DD-decision-{slug}.md`
  - 파일명은 memento의 `decision_note_format` 통일 규칙(2.7.0+)에 따라 `decision-` prefix 포함
  - 동일 파일명 존재 시 slug 뒤에 `-2`, `-3` suffix 자동 부여

**auto memory 저장**:
- 결정 사항을 auto memory `project` 타입으로 저장
- 정의 + 결정 + 핵심 근거 + 날짜

### 페르소나 에이전트 프롬프트 구성

각 Agent 호출 시:
1. 해당 페르소나 파일을 `${CLAUDE_PLUGIN_ROOT}/personas/` 에서 Read
2. 프롬프트에 포함:
   - 페르소나 역할 + 분석 가이드 (파일 내용)
   - **원안 요약 (인용)** — Step 0 결과
   - 주제 + 유형 + 맥락 (Step 0 결과)
   - 이전 단계/라운드 결과 (해당 시)
   - 라운드 주제 + 지시 (Step 2·4의 경우)

### 모델 배정

**haiku 사용 금지** — 모든 Agent 호출에 `model` 파라미터 명시.

| 모델 | 대상 | 이유 |
|------|------|------|
| `opus` | ontologist, sw-expert, architect, linguist | 본질 분석, 체계 매핑, 시스템 사고 |
| `sonnet` | user-proxy, 나머지 벤치, 리서치 에이전트, 검증 에이전트 | 경험 수집, 구조화된 리뷰, 검색/검증 |

### 실행 모델 요약

| 단계 | 실행 모델 | 이유 |
|------|----------|------|
| Step 1 | 단독 (user-proxy) + facilitator Q&A | 경험 수집은 단일 출처 |
| Step 2 | **라운드 기반 상호작용** | 본질 합의는 상호 반박으로 형성 |
| Step 3a | 병렬 (독립) | 리서치 커버리지 |
| Step 3b | 병렬 (독립, 리서치와 분리) | hallucination 방어 |
| Step 4 | **라운드 기반 상호작용** | 다면 검증은 상호 반박으로 형성 |

### facilitator 책임 (메인 세션)

- **발언권 배분**: 각 라운드 발언자 지정
- **컨텍스트 번들링**: 이전 발언 + 원안 + 라운드 주제를 한 프롬프트로 합성
- **라운드 종료 판정**: 수렴 / 교착 / 신규 축 출현
- **교착 해소**: 이견 정리 라운드 또는 사용자 위임
- **정합성 재확인**: Step 1·2·4 종료 시 Step 0d 재체크
- **규모 조정**: mode·bench_policy에 따라 라운드 수 축약/확장
