# Ontology Workshop

정의가 명확하면 이름은 자연스럽게 따라온다. 정의가 모호하면 어떤 투표도 수렴하지 않는다.

## 핵심 원칙

- **투표는 도구이지 목적이 아니다** — 정의의 검증 수단으로만 사용
- **온톨로지가 척추** — 모든 단계가 "이것은 무엇인가?"라는 질문을 중심으로 전개
- **검증 없는 주장은 근거가 아니다** — 리서치는 반드시 교차 검증
- **내부 Task ID 축약 단독 사용 금지** — 워크숍 산출물(결정 기록, 정의 노트)에서 `T1`~`T9`, `CP1`~`CP9`, `KR1`~`KR9` 같은 축약을 한 문서 내 첫 출현 시 풀어쓰거나 괄호 병기. 이후 반복은 단독 허용. 상세: 저장소 CLAUDE.md "사용자 대면 출력 규칙"

## 페르소나 시스템

### 코어 멤버 (항상 참여)

| ID | 역할 | 분석 축 | 프롬프트 파일 |
|----|------|---------|-------------|
| `ontologist` | 온톨로지스트 | 본질 (essence) | `personas/core/ontologist.md` |
| `sw-expert` | SW공학 전문가 | 체계 (taxonomy) | `personas/core/sw-expert.md` |
| `user-proxy` | 사용자 대리인 | 경험 (experience) | `personas/core/user-proxy.md` |

### 벤치 멤버 (주제별 추가)

자동 선택 가이드: `reference/persona-selection.md` 참조.

| 분석 유형 | 자동 추가 |
|----------|----------|
| 용어/네이밍 | linguist, backend-dev, frontend-dev, designer |
| 기술 선택 | architect, backend-dev, devops |
| 아키텍처 | architect, backend-dev, devops, security |
| UI/UX 결정 | designer, frontend-dev, pm |
| 데이터 모델 | architect, data-engineer, backend-dev |

AskUserQuestion으로 사용자에게 벤치 멤버 추가/제거 확인.

## 워크플로우

```
[주제] → 0.파악 → 1.현상학 → 2.온톨로지 → 3.분류학+검증 → 4.다면검증 → 5.도출+기록
                   (경험)     (본질)      (위치)          (수렴)     (결정)
```

---

### Step 0: 주제 파악

1. 사용자 입력에서 분석 대상 파악
2. AskUserQuestion으로 분석 유형 확인 + 맥락 수집:

| 유형 | 예시 | 핵심 질문 |
|------|------|----------|
| 용어/네이밍 | "이걸 영어로 뭐라 부를까?" | 이것은 무엇인가? |
| 개념 정의 | "requirement와 specification 차이가 뭐야?" | 이 둘은 어떻게 다른가? |
| 기술 선택 | "상태 관리 뭘 쓸까?" | 우리 문제의 본질은 무엇인가? |
| 아키텍처 | "이 구조가 맞나?" | 이 시스템의 본질적 제약은 무엇인가? |

3. 유형에 따라 벤치 멤버 자동 추천 → 사용자 확인

**Output**: 분석 대상, 유형, 참여 페르소나 확정

---

### Step 1: 현상학 (Phenomenology) — "경험으로서의 대상"

**목적**: 이론적 분석 전에, 대상이 실제로 어떻게 경험되는지 수집.

**실행**: `user-proxy` 페르소나 프롬프트(`personas/core/user-proxy.md`)를 읽고 Agent로 디스패치.

질문 축:
- "이걸 언제, 어떻게, 왜 쓰는가?"
- "동료에게 뭐라고 말하는가?"
- "화면에서 이걸 보면 뭐라고 부르는가?"
- "이것 없이 일하면 뭐가 불편한가?"

**Output**: 사용자 경험 기술서 — 실제 사용 맥락, 자연어 표현, 일상적 호칭

> 이 단계는 이후 모든 분석의 **현실 앵커** 역할. 이론이 경험에서 벗어나면 이 단계로 돌아옴.

---

### Step 2: 온톨로지 분석 (Ontological Analysis) — "본질"

**목적**: 대상이 무엇이고, 무엇이 아닌지 엄밀하게 정의.

**실행**: 코어 멤버 + 해당 벤치 멤버를 **병렬 Agent로 디스패치**.

각 에이전트에게 전달할 정보:
- Step 0의 주제 + 유형
- Step 1의 경험 기술서
- 해당 페르소나 프롬프트 (`personas/` 에서 Read)

| 분석자 | 질문 | 방법 |
|--------|------|------|
| ontologist | Genus(상위 개념)? Differentia(차별점)? | "X인가?" 질문 체인 + 소거법 |
| sw-expert | 업계 분류 체계에서 어디에 위치? | 표준/학술 체계 매핑 |
| (벤치) architect | 경계, 관계, 생명주기? | 입력/출력/상태 분석 |
| (벤치) linguist | 사용자가 쓰는 단어의 실제 의미 범위? | 의미론적 분석 |

**Output**:
```
정의: "A ___ is a [genus] that [differentia]"
경계: [이것이 아닌 것 목록]
속성: [핵심 속성 목록]
관계: [상위/하위/인접 개념과의 관계]
```

---

### Step 3: 분류학 + 리서치 검증 (Taxonomy & Verification) — "위치"

**목적**: Step 2의 정의를 기존 체계에 매핑하고, 주장을 검증.

#### 3a. 리서치 (Agent 2-3개 병렬)

WebSearch를 사용하여:
- 표준/학계에서 이 개념에 해당하는 기존 용어 조사
- 실제 도구/업계에서 유사 개념을 어떻게 부르는지 조사
- (해당 시) DDD/도메인 언어 패턴 조사

#### 3b. 교차 검증 (리서치와 반드시 다른 Agent)

리서치 결과에서 핵심 주장을 추출한 뒤:

```
[리서치 에이전트 A] → 주장1 + 출처, 주장2 + 출처
[리서치 에이전트 B] → 주장3 + 출처, 주장4 + 출처
         ↓ (핵심 주장 추출)
[검증 에이전트 V1] → 주장1 독립 WebSearch 검증
[검증 에이전트 V2] → 주장2 독립 WebSearch 검증  (병렬 실행)
[검증 에이전트 V3] → 주장3 독립 WebSearch 검증
[검증 에이전트 V4] → 주장4 독립 WebSearch 검증
         ↓
검증 보고서: 각 주장별 ✅/⚠️/❌
```

- **주장 1개 = 검증 에이전트 1개** (병렬 실행)
- 각 검증 에이전트는 해당 주장만 독립적으로 WebSearch하여 확인
- 출처 URL 실재 여부, 인용 정확성, 문맥 왜곡 여부 확인
- **❌ 태깅된 주장은 이후 단계에서 근거로 사용 금지**

**Output**: 검증된 분류 체계 매핑 + 기존 용어 조사 + 검증 보고서

---

### Step 4: 다면 검증 (Multi-Perspective Validation) — "수렴"

**목적**: 정의가 다양한 관점에서 견고한지 검증. (이름 투표가 아님!)

Step 1-3 결과를 벤치 멤버 5-7명에게 공유. 각자 자기 관점에서 **병렬 Agent 디스패치**.

**검토 질문** (정의 중심):
1. "이 정의가 당신의 실무 경험과 맞는가?"
2. "빠진 측면이나 반례가 있는가?"
3. "이 정의를 받아들인다면, 어떤 이름이 자연스러운가?"

**분기**:
- 정의에 이견이 있으면 → 이견 분석 → 정의 수정 → Step 2로 1회 회귀
- 정의에 수렴 → 이름/결정 후보 자연 도출

**이름/결정이 필요한 경우에만**:
- 정의에서 후보가 2개 이상 → 정의와의 정합성 기준 간단 평가
- 1차 평가로 수렴하지 않으면 → 상위 2개로 팀 토론 → 최종 투표 (1회만)

**Output**: 검증된 정의 + (필요 시) 확정된 이름/결정

---

### Step 5: 도출 & 기록 (Derivation & Record) — "결정"

#### 5a. 최종 검증

최종 결과의 핵심 근거를 검증 에이전트로 재검증.

#### 5b. 사용자 확인

AskUserQuestion으로 최종 결과 제시:
- 확정 정의
- 확정 이름/결정
- 근거 요약 (✅ 검증된 것만)
- (해당 시) 평가 결과

#### 5c. 기록

**Obsidian vault 저장**:
- memento config (`~/.claude/plugins/data/memento-cc-plugins/config.md`)에서 vault 경로 읽기
- `templates/decision-record.md` 템플릿 사용
- 저장 위치: `{vault}/12 Records/decisions/YYYY-MM-DD-decision-{slug}.md`
  - 파일명은 memento의 `decision_note_format` 통일 규칙(2.7.0+)에 따라 `decision-` prefix 포함
  - 예: `2026-04-20-decision-qmd-fork-unnecessary.md`
  - 동일 파일명 존재 시 slug 뒤에 `-2`, `-3` suffix 자동 부여

**auto memory 저장**:
- 결정 사항을 auto memory `project` 타입으로 저장
- 정의 + 결정 + 핵심 근거 + 날짜

## 에이전트 디스패치 패턴

### 페르소나 에이전트 프롬프트 구성

각 Agent 호출 시:
1. 해당 페르소나 파일을 `${CLAUDE_PLUGIN_ROOT}/personas/` 에서 Read
2. 프롬프트에 포함:
   - 페르소나 역할 + 분석 가이드 (파일 내용)
   - 분석 대상 + 맥락 (Step 0 결과)
   - 이전 단계 결과 (해당 시)
   - 구체적 질문/지시

### 모델 배정

**haiku 사용 금지** — 모든 Agent 호출에 `model` 파라미터를 반드시 명시한다.

| 모델 | 대상 | 이유 |
|------|------|------|
| `opus` | ontologist, sw-expert, architect, linguist | 본질 분석, 다중 체계 매핑, 시스템 사고 등 깊은 추론 필요 |
| `sonnet` | user-proxy, 나머지 벤치 멤버, 리서치 에이전트, 검증 에이전트 | 경험 수집, 기준 기반 리뷰, 검색/검증 등 구조화된 작업 |

### 병렬 실행 원칙

- 독립적인 분석은 반드시 병렬 Agent로 실행
- 리서치 에이전트와 검증 에이전트는 반드시 분리 (독립성 보장)
- Step 2 코어+벤치 분석: 병렬
- Step 3a 리서치: 병렬 → 3b 검증: 병렬 (리서치 완료 후)
- Step 4 다면 검증: 병렬

### 에이전트 결과 종합

각 단계의 병렬 에이전트 결과를 메인 세션에서 종합:
- 일치하는 결론 → 강화
- 상충하는 관점 → 사용자에게 이견 제시 + 분석
- 검증 실패한 근거 → 제거
