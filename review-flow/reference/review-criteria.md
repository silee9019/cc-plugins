# Review Criteria

review-flow 플러그인의 범용 리뷰 기준. plan-review와 code-review 스킬이 공유한다.

## Plan Review 기준

설계/계획 단계에서 검토할 항목.

### 요구사항 충족

- 사용자의 요구사항을 정확히 반영하는가
- 빠진 요구사항이나 암묵적 가정은 없는가
- 해결하려는 문제가 명확히 정의되어 있는가

### 단순성 (YAGNI)

- 요구사항 대비 과잉 설계가 아닌가
- 불필요한 추상화나 미래 대비 설계가 포함되어 있지 않은가
- 더 단순한 대안이 존재하는가

### 아키텍처 정합성

- 기존 코드베이스의 패턴/컨벤션과 일관되는가
- CLAUDE.md 등 프로젝트 가이드라인에 부합하는가
- 기존 유틸리티/함수를 재사용할 수 있는 부분은 없는가

### 엣지케이스

- 경계 조건을 고려했는가
- 에러/실패 시나리오에 대한 처리 계획이 있는가
- 동시성, 순서 의존성 등 비기능 요구사항을 고려했는가

### 검증 전략

- 어떻게 테스트할 것인지 계획이 있는가
- 성공 기준이 검증 가능한 형태로 정의되어 있는가
- 수동 테스트와 자동 테스트의 범위가 적절한가

## Code Review 기준

코드 변경 시 검토할 6개 관점. pr-review-toolkit의 전문 에이전트 관점을 통합한 것이다.

### 1. 코드 품질

프로젝트 가이드라인 준수와 버그 감지.

- CLAUDE.md 규칙 위반 여부
- 로직 에러, off-by-one, null/undefined 처리
- 보안 취약점 (injection, XSS 등)
- 성능 문제 (불필요한 루프, 메모리 누수)
- 네이밍 일관성, 코드 중복

**confidence ≥80인 이슈만 보고한다.**

### 2. 에러 핸들링

무음 실패와 부적절한 에러 처리 탐지.

- 빈 catch 블록 (절대 금지)
- catch 후 로깅 없이 계속 실행하는 패턴
- 사용자에게 actionable하지 않은 에러 메시지
- 과도하게 넓은 exception 캐치
- 에러를 삼키는 fallback 로직

**심각도: CRITICAL (무음 실패) > HIGH (부적절 에러 메시지) > MEDIUM (컨텍스트 부족)**

### 3. 타입 설계

타입의 캡슐화와 불변성 검증.

- 내부 구현이 적절히 은닉되어 있는가
- 불변식(invariant)이 타입 구조로 표현되는가
- 생성자에서 유효성 검증이 이루어지는가
- 변경 지점(mutation point)이 보호되는가

**새 타입이 추가된 경우에만 적용.**

### 4. 테스트 커버리지

동작 기반 테스트 커버리지와 품질.

- 핵심 비즈니스 로직의 테스트가 존재하는가
- 에러 경로와 엣지케이스가 테스트되는가
- 테스트가 구현이 아닌 동작(behavior)을 검증하는가
- 리팩토링에 취약한 테스트는 없는가

**치명도 1-10 점수 (9-10: 데이터 손실/보안, 7-8: 사용자 영향, 5-6: 엣지케이스)**

### 5. 주석 정확성

코드와 주석의 일치 여부, 기술부채 감지.

- 주석이 실제 코드 동작과 일치하는가
- "왜"를 설명하는 주석이 충분한가
- 명백한 코드를 반복 설명하는 불필요한 주석은 없는가
- TODO/FIXME가 이미 해결된 것은 없는가

**주석 추가/수정이 있는 경우에만 적용.**

### 6. 코드 단순화

불필요한 복잡도와 가독성 문제.

- 중첩 삼항 연산자, 과도한 인라인
- 불필요한 추상화나 래퍼
- 읽기 어려운 한 줄 표현 (one-liner)
- 더 명확한 대안이 있는 패턴

**기능은 반드시 보존하면서 단순화한다.**

## 공통: 심각도 분류

| 등급 | confidence | 의미 | 조치 |
|------|-----------|------|------|
| Critical | 90-100 | 버그, 보안 취약점, CLAUDE.md 위반 | 반드시 수정 |
| Important | 80-89 | 유의미한 품질 이슈 | 수정 권장 |
| Suggestion | 60-79 | 개선 제안 | 선택적 |

**confidence < 80인 이슈는 보고하지 않는다.**

## 공통: 프로젝트별 기준 병합

1. CLAUDE.md에서 프로젝트 가이드라인을 읽는다
2. 린터/포매터 설정 (.eslintrc, pyproject.toml 등)을 참조한다
3. 범용 기준과 프로젝트별 기준이 충돌하면 프로젝트별 기준을 우선한다
