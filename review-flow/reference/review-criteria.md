# Review Criteria

review-flow 플러그인의 범용 리뷰 기준. plan-review, code-review, skill-review 스킬이 공유한다.

## Priority-0: 의도 정합성 (모든 리뷰 스킬 공통)

모든 리뷰는 기술적 품질 평가보다 **먼저** 사용자 의도와 산출물의 정합성을 검증한다. 기술적 완성도가 높더라도 의도를 빗나간 산출물은 BLOCK 처리한다.

- **상세 프로토콜**: `${CLAUDE_PLUGIN_ROOT}/reference/intent-alignment.md`
- **판정**:
  - **Clear & Aligned** → 아래 기술적 기준을 이어서 적용
  - **Clear but Misaligned** → 일반 리뷰 중단. 정합성 복구 항목을 Fix Plan으로 먼저 사용자에게 컨펌
  - **Unclear** → 회의 주최 (AskUserQuestion 또는 `knowledge-tools:ontology-workshop` 위임) 후 재실행

**아래의 기술적 기준은 "Clear & Aligned" 판정을 받은 경우에만 적용된다.**

## Plan Review 기준

설계/계획 단계에서 검토할 항목.

### 요구사항 충족

- 사용자의 요구사항을 정확히 반영하는가
- 빠진 요구사항이나 암묵적 가정은 없는가
- 해결하려는 문제가 명확히 정의되어 있는가

### 단순성 (YAGNI)

- 요구사항 대비 과잉 설계가 아닌가
- 불필요한 추상화나 미래 대비 설계가 포함되어 있지 않은가
- 더 단순한 대안이 존재하는가

### 아키텍처 정합성

- 기존 코드베이스의 패턴/컨벤션과 일관되는가
- CLAUDE.md 등 프로젝트 가이드라인에 부합하는가
- 기존 유틸리티/함수를 재사용할 수 있는 부분은 없는가

### 엣지케이스

- 경계 조건을 고려했는가
- 에러/실패 시나리오에 대한 처리 계획이 있는가
- 동시성, 순서 의존성 등 비기능 요구사항을 고려했는가

### 검증 전략

- 어떻게 테스트할 것인지 계획이 있는가
- 성공 기준이 검증 가능한 형태로 정의되어 있는가
- 수동 테스트와 자동 테스트의 범위가 적절한가

## Code Review 기준

코드 변경 시 검토할 6개 관점. pr-review-toolkit의 전문 에이전트 관점을 통합한 것이다.

### 1. 코드 품질

프로젝트 가이드라인 준수와 버그 감지.

- CLAUDE.md 규칙 위반 여부
- 로직 에러, off-by-one, null/undefined 처리
- 보안 취약점 (injection, XSS 등)
- 성능 문제 (불필요한 루프, 메모리 누수)
- 네이밍 일관성, 코드 중복

**confidence ≥80인 이슈만 보고한다.**

### 2. 에러 핸들링

무음 실패와 부적절한 에러 처리 탐지.

- 빈 catch 블록 (절대 금지)
- catch 후 로깅 없이 계속 실행하는 패턴
- 사용자에게 actionable하지 않은 에러 메시지
- 과도하게 넓은 exception 캐치
- 에러를 삼키는 fallback 로직

**심각도: CRITICAL (무음 실패) > HIGH (부적절 에러 메시지) > MEDIUM (컨텍스트 부족)**

### 3. 타입 설계

타입의 캡슐화와 불변성 검증.

- 내부 구현이 적절히 은닉되어 있는가
- 불변식(invariant)이 타입 구조로 표현되는가
- 생성자에서 유효성 검증이 이루어지는가
- 변경 지점(mutation point)이 보호되는가

**새 타입이 추가된 경우에만 적용.**

### 4. 테스트 커버리지

동작 기반 테스트 커버리지와 품질.

- 핵심 비즈니스 로직의 테스트가 존재하는가
- 에러 경로와 엣지케이스가 테스트되는가
- 테스트가 구현이 아닌 동작(behavior)을 검증하는가
- 리팩토링에 취약한 테스트는 없는가

**치명도 1-10 점수 (9-10: 데이터 손실/보안, 7-8: 사용자 영향, 5-6: 엣지케이스)**

### 5. 주석 정확성

코드와 주석의 일치 여부, 기술부채 감지.

- 주석이 실제 코드 동작과 일치하는가
- "왜"를 설명하는 주석이 충분한가
- 명백한 코드를 반복 설명하는 불필요한 주석은 없는가
- TODO/FIXME가 이미 해결된 것은 없는가

**주석 추가/수정이 있는 경우에만 적용.**

### 6. 코드 단순화

불필요한 복잡도와 가독성 문제.

- 중첩 삼항 연산자, 과도한 인라인
- 불필요한 추상화나 래퍼
- 읽기 어려운 한 줄 표현 (one-liner)
- 더 명확한 대안이 있는 패턴

**기능은 반드시 보존하면서 단순화한다.**

## 공통: 심각도 분류

| 등급 | confidence | 의미 | 조치 |
|------|-----------|------|------|
| Critical | 90-100 | 버그, 보안 취약점, CLAUDE.md 위반 | 반드시 수정 |
| Important | 80-89 | 유의미한 품질 이슈 | 수정 권장 |
| Suggestion | 60-79 | 개선 제안 | 선택적 |

**confidence < 80인 이슈는 보고하지 않는다.**

## 공통: 프로젝트별 기준 병합

1. CLAUDE.md에서 프로젝트 가이드라인을 읽는다
2. 린터/포매터 설정 (.eslintrc, pyproject.toml 등)을 참조한다
3. 범용 기준과 프로젝트별 기준이 충돌하면 프로젝트별 기준을 우선한다

## 공통: 수정 계획(Fix Plan) 필수 포함

모든 리뷰 스킬(plan-review / code-review / skill-review)의 보고서는 "발견 사항 나열"로 끝나지 않고 반드시 **실행 가능한 수정 계획** 섹션을 포함한다. 사용자는 이 계획을 보고 **컨펌 / 피드백 / 부분 승인** 중 하나를 선택하며, 선택에 따라 리뷰 스킬이 후속 동작을 수행한다.

보고서 말미 공통 템플릿:

```markdown
## 수정 계획 (Fix Plan)

> 이 계획에 대해 **컨펌**하시면 위에서 아래 순서로 수정을 진행합니다. 수정이 필요하면 **피드백**을 주세요.

### 필수 수정 (Critical + Important)

#### 1. {피드백 요약 제목 — 어떤 문제/개선인지 한 줄. 파일명 대신 주제}

- **피드백 내용**: {현재 코드/설계/문서가 어떤 상태인지 + 위치(`path/to/file.ext:line` 또는 섹션)}
- **왜 제안했는지**: {발생 가능한 문제 · 위반한 원칙 · 근거가 되는 가이드라인}
- **어떻게 수정하면 좋은지**: {구체적 액션(필요 시 코드/패치 스니펫) + 수정 후 검증 방법}
- **사용자 의도와의 충돌 여부**: *(조건부 — 수집된 사용자 의도·plan·CLAUDE.md 원칙 중 하나라도 이 제안과 상충할 여지가 있을 때만 기술. 없으면 이 bullet 자체를 생략.)* {충돌 지점과 상충 의도의 출처, 충돌임에도 제안한 이유 또는 사용자가 의도를 재확인해야 할 지점}

#### 2. {다음 피드백 요약 제목}

- **피드백 내용**: …
- **왜 제안했는지**: …
- **어떻게 수정하면 좋은지**: …
- **사용자 의도와의 충돌 여부**: *(해당 시)* …

### 선택 수정 (Suggestion)

#### 1. {피드백 요약 제목}

- **피드백 내용**: …
- **왜 제안했는지**: …
- **어떻게 수정하면 좋은지**: …
- **사용자 의도와의 충돌 여부**: *(해당 시)* …

### 사용자 액션 요청
- [ ] **컨펌** → 위 계획대로 수정 진행
- [ ] **피드백** → 아래 피드백을 받아 계획 수정 후 재확인
- [ ] **부분 승인** → 특정 항목만 선택하여 수정
```

포맷 규칙:
- 각 피드백은 번호 매겨진 `####` 서브헤딩, 제목은 파일명·섹션명이 아닌 **주제 요약 문장**
- 3개 고정 bullet(**피드백 내용 / 왜 제안했는지 / 어떻게 수정하면 좋은지**) + **조건부 4번째 bullet(사용자 의도와의 충돌 여부)**
- Step 0 매핑 테이블 형식(요구사항-매핑)은 Fix Plan 내부에서 재사용하지 않는다 — 의도 정합성은 Step 0 섹션에서만 표현
- "실행 순서" 별도 섹션은 두지 않는다 — 각 피드백의 번호가 곧 실행 순서

각 스킬은 보고서 출력 직후 `AskUserQuestion`으로 세 선택지를 제시하고, 선택에 따라 다음을 수행한다:

- **컨펌** → 필수 수정부터 순차 실행, 각 수정 후 검증 요약 보고
- **피드백** → 피드백 반영하여 Fix Plan 재작성 후 다시 사용자 컨펌 요청
- **부분 승인** → 선택된 항목만 실행

`intent-alignment.md`의 "Clear but Misaligned → BLOCK" 케이스에서도 이 Fix Plan 형식으로 "정합성 복구 항목"을 우선 제시하여 사용자 컨펌을 받는다.
