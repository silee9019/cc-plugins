# Intent Alignment — Priority-0 공유 기준

review-flow의 모든 리뷰 스킬(plan-review / code-review / skill-review)이 **기술적 품질 평가 전에 가장 먼저** 적용하는 공유 기준. 기술적 완성도보다 "사용자의 의도/요구사항을 정확히 반영했는가"가 우선이다.

## 핵심 원칙

> "설계/코드/문서가 사용자의 의도/요구사항을 정확히 반영하고 있는가?"

기술적 기준(코드 품질, 에러 핸들링, frontmatter 유효성 등)은 이 질문에 "예"가 나온 경우에만 적용된다. "아니오" 또는 "불분명"이면 일반 리뷰를 중단하고 이 문서의 프로토콜을 따른다.

## 적용 절차

### 1. 의도 출처 수집

다음 세 출처를 모두 확인한다.

1. **현재 대화 컨텍스트** — 사용자가 이번 세션에서 직접 표현한 요구사항, 제약, 선호
2. **문서화된 요구사항** — CLAUDE.md, README.md, 관련 이슈/PR 설명, 플랜 파일, 커밋 메시지
3. **프로젝트 규칙/컨벤션** — 프로젝트 고유 가이드라인, 네이밍, 아키텍처 원칙

### 2. 의도-산출물 매핑

각 요구사항마다 산출물(설계/코드/문서)에서 대응되는 부분을 찾아 매핑한다.

| 매핑 결과 | 해석 |
|----------|------|
| 요구사항 → 산출물 대응부 존재 | ✅ 반영 |
| 요구사항 → 대응부 없음 | ❌ **누락** |
| 산출물에 있으나 요구사항에 없음 | ⚠️ **과잉 / 자의적 추가** |

### 3. 판정

매핑 결과에 따라 세 가지 판정을 내린다.

| 판정 | 조건 | 조치 |
|------|------|------|
| **Clear & Aligned** | 의도가 명확하고 산출물이 이를 정확히 반영 (누락/과잉 없음) | 일반 리뷰(기술적 기준)로 진행 |
| **Clear but Misaligned** | 의도는 명확하나 산출물이 빗나감 (누락 또는 과잉 존재) | **BLOCK**. 일반 리뷰 중단. 수정 권장 항목을 Fix Plan 형태로 제시하여 사용자 컨펌을 먼저 받음 |
| **Unclear** | 의도/요구사항 자체가 불분명 | **회의 주최**. 일반 리뷰 중단. 아래 프로토콜 수행 |

> ※ 각 스킬의 **총평 매핑**(PASS/WARN/BLOCK 임계, BLOCK 확정 조건 등)은 이 표가 아니라 해당 스킬 정의(SKILL.md의 Step 6 등)를 우선한다. 이 문서는 의도 정합성 판정 프로토콜만 정의한다.

### 4. 회의 주최 프로토콜 (Unclear인 경우)

의도가 불분명하면 리뷰를 이어가지 않고 명확화부터 수행한다.

- **간단한 불명확** (단일 선택지, 구체적 확인 질문) → `AskUserQuestion`으로 사용자에게 직접 확인
- **복잡한 불명확** (용어 정의, 개념 구별, 다중 대안 비교, 의사결정 필요) → `knowledge-tools:ontology-workshop` 스킬로 위임

회의 결과로 의도가 명확해지면 Step 0부터 재실행한다. 명확해지지 않으면 리뷰를 보류하고 사용자에게 보류 사유를 보고한다.

## 보고서 섹션 템플릿

모든 리뷰 스킬은 보고서의 "총평" 바로 다음에 아래 섹션을 포함한다.

```markdown
## Step 0: 의도 정합성 검증

### 의도/요구사항 (수집됨)
1. [요구사항 1] — 출처: [대화 / 문서 / 컨벤션]
2. [요구사항 2] — 출처: [...]

### 산출물 매핑
| 요구사항 | 산출물 매핑 | 상태 |
|----------|------------|------|
| 요구사항 1 | [대응 부분] | ✅ 반영 |
| 요구사항 2 | — | ❌ 누락 |
| — | [산출물 부분] | ⚠️ 과잉 |

### 판정: Clear & Aligned | Clear but Misaligned | Unclear
- Clear & Aligned → 아래 일반 리뷰 진행
- Clear but Misaligned → 일반 리뷰 중단. 아래 Fix Plan의 "정합성 복구 항목" 우선 컨펌 요청
- Unclear → 회의 주최 (방식: AskUserQuestion | ontology-workshop 위임)
```

## 스킬별 의도 수집 포커스

| 스킬 | "의도"가 의미하는 것 |
|------|---------------------|
| plan-review | 설계/계획이 해결해야 하는 문제, 사용자가 원하는 결과 |
| code-review | 이번 커밋/변경이 반영해야 하는 요청, 관련 이슈/PR의 목표 |
| skill-review | 이 스킬/커맨드가 의도한 트리거 조건, 실행 동작, 범위, 제약 |
